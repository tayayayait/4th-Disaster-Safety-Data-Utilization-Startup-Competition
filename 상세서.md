# 침수퇴로 AI UI/UX 상세서

문서 상태: MVP 설계 기준  
대상 제품: 침수퇴로 AI  
대상 플랫폼: 모바일 PWA 우선, 데스크톱 담당자 대시보드 병행  
기준 해상도: 모바일 360px 이상, 태블릿 768px 이상, 데스크톱 1280px 이상  
작성 목적: 개발자와 디자이너가 별도 해석 없이 동일한 UI/UX 기준으로 MVP를 구현하기 위한 상세 명세

## 1. 제품 정의

침수퇴로 AI는 집중호우, 하천범람, 해안침수, 지하차도 침수 위험 상황에서 사용자의 현재 위치를 기준으로 안전한 대피소와 차량/도보 퇴로를 추천하는 재난 대응 서비스다.

핵심 가치는 다음 3가지다.

- 사용자는 "지금 이동해야 하는지, 기다려야 하는지"를 즉시 판단한다.
- 사용자는 "가장 빠른 길"이 아니라 "침수·범람·통제 위험이 낮은 길"을 선택한다.
- 담당자는 위험지역, 통제 후보, 주민 안내문, 확인 대상 데이터를 한 화면에서 확인한다.

## 2. MVP 범위

### 2.1 포함 범위

| 구분        | MVP 포함 내용                                                    |
| ----------- | ---------------------------------------------------------------- |
| 사용자 유형 | 시민, 차량 운전자, 지자체 담당자                                 |
| 공간 범위   | 침수 취약 생활권 1개 구역                                        |
| 대피소      | 민방위대피시설 또는 지자체 대피시설 20~30개                      |
| 위험 레이어 | 침수흔적도 WMS, 하천범람지도 WMS, 기상청 강우/예보, 긴급재난문자 |
| 차량 경로   | NAVER Cloud Maps Directions 5                                    |
| 도보 경로   | TMAP 보행자 경로안내                                             |
| AI          | Gemini 기반 재난문자 분류, 경로 설명, 담당자 조치문 생성         |
| 관리자      | 위험지역 TOP 5, 통제 후보, API 수집 상태, 주민 안내문 초안       |

### 2.2 제외 범위

| 제외 항목             | 제외 사유                                                             |
| --------------------- | --------------------------------------------------------------------- |
| 전국 실시간 서비스    | MVP 일정상 데이터 정합성 검증 범위가 과도함                           |
| 실제 수위 센서 연동   | 지역별 센서 접근권한과 품질 차이가 큼                                 |
| 자동 대피명령 발령    | 법적·행정적 책임 범위가 MVP를 초과함                                  |
| 사용자 위치 상시 추적 | 개인정보 위험이 큼. MVP는 명시적 위치 사용만 허용                     |
| Gemini 단독 위험 판단 | 위험도는 서버 계산값을 기준으로 하고 Gemini는 설명·문안 생성에만 사용 |

## 3. 정보 구조

### 3.1 시민용 앱 내비게이션

모바일 하단 탭은 4개로 제한한다.

| 탭  | 라벨   | 주요 기능                    |
| --- | ------ | ---------------------------- |
| 1   | 홈     | 현재 위험도, 추천 행동, 지도 |
| 2   | 경로   | 도보/차량 경로 비교          |
| 3   | 대피소 | 대피소 목록, 상세, 수용정보  |
| 4   | 도움   | AI 질문, 행동요령, 공유      |

탭 높이는 64px, 아이콘 22px, 라벨 11px로 고정한다. 심각 상태에서는 하단 탭 위에 긴급 행동 바를 추가한다.

### 3.2 담당자용 대시보드 내비게이션

데스크톱은 좌측 사이드바를 사용한다.

| 메뉴        | URL 예시           | 설명                      |
| ----------- | ------------------ | ------------------------- |
| 상황판      | `/ops`             | 위험지역, 지도, 통제 후보 |
| 위험지역    | `/ops/risk-zones`  | TOP 위험지역 목록         |
| 대피소      | `/ops/shelters`    | 수용현황, 운영상태        |
| 알림문안    | `/ops/messages`    | 주민 안내문, 기관 연락문  |
| 데이터 상태 | `/ops/data-health` | API 상태, 최신 수집시각   |

사이드바 너비는 248px, 접힘 상태는 72px로 한다.

## 4. 사용자 흐름

### 4.1 시민: 현재 위치에서 대피 여부 판단

1. 사용자가 홈 화면에 진입한다.
2. 위치 권한 요청 모달을 표시한다.
3. 위치 허용 시 현재 위치 기준 2km 반경의 위험도와 대피소를 조회한다.
4. 시스템은 위험도를 `안전`, `주의`, `경계`, `심각` 중 하나로 산출한다.
5. 홈 하단 행동 카드에 추천 행동을 표시한다.
6. 사용자는 `도보 경로 보기` 또는 `차량 경로 보기`를 선택한다.
7. 경로 화면에서 추천 경로, 대안 경로, 제외 경로를 비교한다.
8. 사용자는 경로를 공유하거나 AI에게 추가 질문한다.

완료 조건:

- 현재 위험도 표시까지 3초 이내
- 추천 대피소 1~3개 표시
- 데이터 기준시각 표시
- 위치 거부 시 주소 입력 대체 흐름 제공

### 4.2 시민: 위치 권한을 거부한 경우

1. 위치 권한 거부 상태를 감지한다.
2. 주소 검색 입력창을 표시한다.
3. 사용자가 주소 또는 장소명을 입력한다.
4. Geocoding 결과를 선택한다.
5. 선택 좌표 기준으로 동일한 위험도·대피소·경로 계산을 수행한다.

위치 거부 화면 문구:

```text
현재 위치를 사용할 수 없습니다.
주소를 입력하면 해당 위치 기준으로 대피 경로를 계산합니다.
```

금지 문구:

```text
위치 권한이 없으면 서비스를 사용할 수 없습니다.
```

### 4.3 시민: 심각 상태 긴급 대피

1. 위험도가 `심각`이면 홈 상단에 전체폭 경고 배너를 표시한다.
2. 하단 행동 카드는 붉은색 배경의 `즉시 대피 권고` 상태가 된다.
3. 기본 CTA는 `가장 안전한 경로 시작`으로 변경한다.
4. 경로 선택 화면에서는 추천 경로만 먼저 표시하고, 대안 경로는 접힌 영역에 둔다.
5. 음성 안내 버튼을 노출한다.

심각 상태에서는 사용자의 선택 부담을 줄이기 위해 한 화면에 1개 주요 행동만 노출한다.

### 4.4 담당자: 위험지역 확인 및 안내문 생성

1. 담당자가 대시보드에 접속한다.
2. 상황판은 기본적으로 위험도 높은 순서 TOP 5를 표시한다.
3. 담당자는 특정 위험지역을 선택한다.
4. 우측 패널에 위험요인, 영향 대피소, 통제 후보 도로를 표시한다.
5. `안내문 생성` 버튼을 누르면 Gemini가 주민 안내문 초안을 생성한다.
6. 담당자는 문안을 수정한 뒤 복사 또는 다운로드한다.

완료 조건:

- TOP 5 위험지역 로딩 5초 이내
- 안내문 초안 생성 8초 이내
- 안내문에는 지역명, 행동요령, 기준시각이 반드시 포함됨

### 5.4 AI 안내 화면

경로: `/help`

#### 입력 방식

자유 입력보다 고정 질문 버튼을 우선한다. 재난 상황에서 사용자의 입력 부담을 줄이기 위한 규칙이다.

고정 질문:

- 지금 나가도 되나요?
- 도보와 차량 중 무엇이 안전한가요?
- 가족에게 보낼 문구를 만들어주세요.
- 대피소까지 위험한 구간이 있나요?
- 담당자에게 신고할 내용을 정리해주세요.

#### 답변 카드 구조

| 영역     | 내용                                    |
| -------- | --------------------------------------- |
| 판단     | 이동/대기/차량 이동/도보 이동           |
| 이유     | 최대 3개 bullet                         |
| 근거     | 기상, 침수흔적, 경로위험, 재난문자      |
| 기준시각 | `YYYY-MM-DD HH:mm`                      |
| 주의     | AI 답변은 공식 대피명령을 대체하지 않음 |

AI 답변에는 출처 없는 도로명, 대피소명, 기관명을 생성하면 안 된다. 서버에서 제공한 데이터 목록 밖의 고유명사가 나오면 `검증 실패` 상태로 처리한다.

## 11. 반응형 기준

### 11.1 Breakpoints

| 이름    | 범위        | 기준 UI                     |
| ------- | ----------- | --------------------------- |
| mobile  | 360~767px   | 하단 탭, 하단 sheet         |
| tablet  | 768~1023px  | 좌측 목록 + 지도            |
| desktop | 1024px 이상 | 사이드바 + 지도 + 상세패널  |
| wide    | 1440px 이상 | 지도 중심, 패널 최대폭 고정 |

### 11.2 모바일 규칙

- 지도는 항상 첫 화면의 50% 이상을 차지한다.
- 하단 행동 카드는 접힌 상태 156px, 펼친 상태 최대 화면 높이 70%.
- CTA는 화면 하단에서 엄지 접근 가능한 위치에 둔다.
- 2열 버튼은 각각 최소 너비 0, gap 8px, 텍스트 줄바꿈 허용.

### 11.3 데스크톱 규칙

- 담당자 화면은 `248px / 1fr / 360px` 3단 구조를 기본으로 한다.
- 브라우저 너비가 1180px 미만이면 우측 상세패널은 drawer로 전환한다.
- 표는 화면 높이 초과 시 내부 스크롤을 사용하고 페이지 전체 스크롤을 피한다.

## 12. 접근성 원칙

### 12.1 색상 대비

- 본문 텍스트 대비는 WCAG AA 기준 4.5:1 이상.
- 18px 이상 굵은 텍스트는 3:1 이상.
- 위험도는 색상만으로 표현하지 않고 라벨과 아이콘을 함께 사용한다.

### 12.2 키보드 접근

- 모든 버튼, 탭, 입력창, 모달 닫기 버튼은 Tab 접근 가능.
- 지도 위 주요 컨트롤은 키보드 focus 가능.
- 모달이 열리면 focus trap 적용.
- ESC는 모달 닫기. 단, 심각 상태 긴급 안내 모달은 닫기 전 확인 버튼을 요구한다.

### 12.3 스크린리더 문구

위험도 배지 예시:

```text
현재 침수 위험도는 경계입니다. 추천 행동은 안전한 대피소 이동 준비입니다.
```

경로 버튼 예시:

```text
도보 추천 경로 보기. 예상 시간 14분, 안전점수 86점.
```

지도 마커는 `aria-label`을 지정한다.

```text
대피소: ○○초등학교, 거리 650미터, 도보 9분
```

### 12.4 재난 상황 가독성

- 심각 상태 핵심 문구는 최소 20px 이상.
- 긴급 CTA는 아이콘만 사용 금지. 반드시 텍스트 포함.
- 진동, 소리, 시각 경고를 각각 독립적으로 끌 수 있어야 한다.

### 14.1 기준시각 표시

```text

```

## 15. 위험도 계산 UI 규칙

### 15.1 위험요인 표시 우선순위

위험요인은 최대 3개만 노출한다.

우선순위:

1. 현재 강우/예보 위험
2. 침수흔적 중첩
3. 하천범람지도 중첩
4. 재난문자 해당지역
5. 지하차도 또는 저지대 통과
6. 교통통제 또는 혼잡

### 15.2 점수 표시

일반 시민 화면에서는 세부 수식 대신 0~100 안전점수와 설명을 보여준다.

담당자 화면에서는 세부 점수 항목을 표시한다.

예:

| 항목          | 점수 |
| ------------- | ---: |
| 강우 위험     |   18 |
| 침수흔적 중첩 |   24 |
| 하천범람 중첩 |   12 |
| 지하차도 통과 |    5 |
| 총 위험점수   |   59 |

## 16. Gemini 적용 UI 규칙

### 16.1 Gemini 답변 생성 조건

Gemini는 다음 데이터가 모두 있을 때만 경로 설명을 생성한다.

- 추천 경로 ID
- 추천 대피소 ID
- 위험도 상태
- 경로 위험요인 1개 이상
- 데이터 기준시각

누락 시 Gemini 호출 금지. 규칙 기반 문구를 사용한다.

### 16.2 Gemini 출력 검증

프론트에 표시하기 전 서버에서 다음 검증을 수행한다.

| 검증                               | 실패 처리          |
| ---------------------------------- | ------------------ |
| JSON schema 일치                   | 폐기               |
| 대피소 ID 존재                     | 폐기               |
| 도로명/지역명이 입력 데이터에 존재 | 불명확 문구로 대체 |
| 행동이 허용 enum에 포함            | fallback           |

허용 행동 enum:

```json
[
  "WAIT",
  "WALK_TO_SHELTER",
  "DRIVE_TO_SAFE_ZONE",
  "AVOID_ROUTE",
  "CALL_119",
  "CHECK_OFFICIAL_NOTICE"
]
```

### 16.3 AI 면책 문구

AI 답변 카드 하단에는 다음 문구를 12px로 표시한다.

```text
AI 안내는 공공데이터와 현재 입력값을 기반으로 한 보조 판단입니다. 지자체 대피명령과 현장 통제를 우선하세요.
```

### 17.2 로딩 상태

| 화면   | skeleton 기준                             |
| ------ | ----------------------------------------- |

| 대피소 | 목록 5개 skeleton                         |
| 담당자 | 지도 skeleton + 표 5행 skeleton           |

## 19. QA 체크리스트

### 19.1 시민 화면
