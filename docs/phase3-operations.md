# Phase 3 운영 기능 정리

## 계정과 권한

- `profiles` 테이블은 `citizen`, `operator`, `admin` 역할을 가진다.
- `/ops` 하위 화면은 공개 현장정보 화면으로 사용하며 시민도 접근할 수 있다.
- 현재 하단 `현장` 탭은 화면 진입 즉시 전국 CCTV 위치를 표시하는 화면(`/ops/cctv`)만 노출한다.
- 실제 상태 변경, 발송, 삭제처럼 외부 효과가 있는 기능은 DB RLS와 서버 함수에서 별도 권한을 확인해야 한다.
- 실제 Supabase DB에 마이그레이션을 적용한 뒤 타입은 `supabase gen types`로 재생성해야 한다.

## 대피소 운영상태

- `shelter_operations` 테이블은 대피소명, 주소, 좌표, 수용인원, 운영상태, 출처, 최신 확인 시각을 저장한다.
- 초기 마이그레이션은 발표용 mock 대피소 12개를 seed로 넣는다.
- 최신 확인 시각이 없거나 7일을 초과하면 UI에서 `CHECK_REQUIRED`로 표시한다.
- 운영상태 조회는 공개 화면에 노출할 수 있다. 운영상태 변경은 RLS 기준으로 권한이 필요하다.

## 알림과 센서

- 푸시 알림은 사용자가 버튼으로 명시 동의한 뒤에만 브라우저 권한을 요청한다.
- 백그라운드 위치 추적은 MVP 범위 밖이며 코드와 DB 제약 모두에서 비활성으로 고정한다.
- 실시간 센서 연동은 실제 데이터 접근권한이 확인되기 전까지 `PENDING_ACCESS` 또는 `DISABLED` 상태로만 표시한다.

## 감사 로그와 모니터링

- 안내문 생성, 현장정보 조치, 대피소 상태 변경은 `audit_logs`에 기록할 수 있다.
- API 상태 스냅샷은 `api_health_metrics`에 저장하며 실패율, fallback 비율, 평균 응답시간을 현장정보 화면에서 확인한다.
- 사용자에게는 일반화된 실패 문구만 보여주고, 로그에도 API key, token, secret 문자열은 제거한다.
